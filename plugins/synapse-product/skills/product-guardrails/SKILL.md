# Synapse 제품 가드레일 (Product Guardrails)

제품 기획자 관점에서 시냅스의 기능 설계, 변경, 확장에 대한 가드레일과 가이드라인을 정의한다.

## 핵심 제품 원칙

### 1. All-in-One MLOps

시냅스는 데이터 업로드부터 모델 배포까지 **단일 플랫폼**에서 완결되어야 한다.

- **DO**: 기존 워크플로우 안에서 자연스럽게 연결되는 기능 추가
- **DON'T**: 별도의 외부 도구 없이는 완료할 수 없는 워크플로우 생성
- **DON'T**: 시냅스 외부에서만 수행해야 하는 중간 단계 요구

### 2. 라벨링 최적화 우선

시냅스의 핵심 경쟁력은 **어노테이션 품질과 효율성**이다.

- **DO**: 라벨링 작업 속도와 정확도를 높이는 기능에 우선순위
- **DO**: AI 보조 기능으로 반복 작업 자동화
- **DON'T**: 라벨링 워크플로우를 복잡하게 만드는 기능 추가
- **DON'T**: 작업자가 어노테이터 외부로 나가야 하는 흐름 설계

### 3. No/Low/High Code 지원

모든 수준의 사용자가 활용할 수 있어야 한다.

- **DO**: 기본 기능은 코딩 없이 UI에서 완료 가능
- **DO**: 고급 사용자를 위해 SDK/API로 확장 가능
- **DO**: 서드파티 플러그인으로 중간 수준 커스터마이징 지원
- **DON'T**: 기본 워크플로우에 코딩 지식 필요
- **DON'T**: SDK 전용 기능이 UI 대비 우선시되는 구조

### 4. 보안 우선 (Security First)

데이터 보안은 타협 불가 원칙이다.

- **DO**: 사용자 지정 스토리지에서 데이터 처리
- **DO**: 역할 기반 접근 제어 (RBAC) 일관 적용
- **DO**: 최소 권한 원칙 (Principle of Least Privilege)
- **DO**: 폐쇄망 환경 지원
- **DON'T**: 데이터가 사용자 통제 밖으로 유출되는 구조
- **DON'T**: 인증/인가 우회 경로 허용
- **DON'T**: 민감 데이터를 로그/캐시에 평문 저장

### 5. 플러그인 확장성

핵심 기능은 플랫폼이, 커스텀은 플러그인이 담당한다.

- **DO**: 핵심 기능은 플랫폼에 내장 (어노테이터, 프로젝트 관리, 데이터 컬렉션)
- **DO**: 도메인 특화 기능은 플러그인으로 분리
- **DO**: 플러그인 API 안정성 유지 (하위 호환)
- **DON'T**: 플러그인으로 핵심 기능을 대체하는 구조
- **DON'T**: 플러그인 없이 기본 워크플로우가 불완전한 상태

## UX 설계 원칙

### 카드형 UI 일관성

- 목록 페이지는 카드형 UI 패턴을 기본으로 사용
- 카드에 썸네일 + 핵심 정보를 함께 표시
- 클릭 없이도 전체 흐름을 파악할 수 있는 정보 밀도
- 새 목록 페이지 설계 시 기존 카드 패턴과 일관성 유지

### Interactive-First

- 모든 사용자 대면 기능은 인터랙티브 우선 설계
- 필수 파라미터가 없을 때 에러 대신 대화형 안내
- 단계별 마법사 패턴 지원
- 고급 사용자를 위한 직접 입력/CLI 옵션도 병행

### 점진적 공개 (Progressive Disclosure)

- 기본 설정은 간단하게, 고급 설정은 필요 시 노출
- 작업자에게는 라벨링에 집중할 수 있는 최소 인터페이스
- 관리자에게는 전체 현황을 파악할 수 있는 대시보드
- 개발자에게는 SDK/API 접근 경로

### 상태 가시성 (Status Visibility)

- 모든 장기 실행 작업 (학습, 익스포트, 업로드)의 진행 상태 표시
- 대시보드에서 프로젝트/워크샵 전체 현황 한눈에 파악
- 오류 발생 시 명확한 메시지와 복구 가이드

## 기능 범위 경계

### 시냅스의 핵심 범위 (In-Scope)

| 영역 | 범위 |
|------|------|
| 데이터 관리 | 업로드, 컬렉션, 파일 스펙, 메타데이터, 버전 관리 |
| 어노테이션 | 이미지/비디오/텍스트/3D/프롬프트 라벨링 |
| 프로젝트 관리 | 워크샵, 태스크, 작업자, 검수, 진행 관리 |
| 품질 관리 | 자동 검토, 검수 워크플로우, 튜토리얼 |
| ML 파이프라인 | 실험, 학습, 모델 관리, 배포 |
| 데이터 익스포트 | JSON, COCO, YOLO, VOC, CSV 변환 |
| 보안·인프라 | 스토리지 연동, RBAC, 폐쇄망 지원 |
| 확장성 | 플러그인 시스템, SDK, API |

### 시냅스의 범위 밖 (Out-of-Scope)

| 영역 | 이유 |
|------|------|
| 범용 데이터 웨어하우스 | 시냅스는 학습 데이터 전문, 범용 DW가 아님 |
| 모델 서빙 인프라 (독립) | 모델 배포는 지원하나, 독립적인 서빙 인프라 제공은 범위 밖 |
| 데이터 크롤링/수집 | 외부 데이터 수집 도구는 범위 밖 — 업로드만 지원 |
| BI/분석 대시보드 (범용) | 어노테이션 관련 대시보드만 제공, 범용 BI 도구 아님 |
| IDE/개발 환경 | SDK를 제공하나, 개발 환경 자체를 제공하지 않음 |
| 작업자 급여/인사 관리 | 작업 할당·관리만 제공, HR 시스템이 아님 |

## 금지 패턴 (Anti-Patterns)

### 아키텍처

- **거대한 단일 API**: 하나의 엔드포인트에서 모든 것을 처리하는 구조
- **클라이언트 과부하**: 프론트엔드에서 대량 데이터 처리를 요구하는 구조
- **API 우회**: 백엔드 API를 우회하여 직접 DB에 접근하는 패턴
- **무한 페이지네이션**: 10K+ 데이터를 API 페이지네이션으로 순차 조회 (BaseExporter 사용)

### UX

- **설정 폭발**: 한 화면에 20개 이상의 설정 옵션 노출
- **컨텍스트 전환**: 하나의 작업을 완료하기 위해 3개 이상의 페이지 이동 필요
- **암묵적 전제**: 사용자가 시스템 내부 구조를 알아야 사용 가능한 인터페이스
- **피드백 부재**: 작업 결과나 오류에 대한 시각적 피드백 없음

### 데이터

- **스키마 무시**: data_unit_meta_schema를 정의했는데 유효성 검증을 건너뛰는 경우
- **데이터 손실**: 포맷 변환 시 원본 데이터의 무손실 보존 경로가 없는 경우
- **하드코딩 포맷**: 새로운 어노테이션 타입이 추가될 수 없는 고정 구조

## 변경 검증 체크리스트

새로운 기능이나 변경을 기획/리뷰할 때 아래 체크리스트로 검증한다.

### 제품 적합성

- [ ] 이 기능은 시냅스의 핵심 범위(In-Scope) 내에 있는가?
- [ ] 기존 End-to-End 워크플로우와 자연스럽게 연결되는가?
- [ ] No/Low/High Code 중 어느 레벨에서 사용 가능한가?

### 사용자 영향

- [ ] 타겟 사용자 역할이 명확한가? (관리자, 작업자, ML 엔지니어 등)
- [ ] 기존 사용자의 워크플로우를 방해하지 않는가?
- [ ] 새 기능을 발견하고 배우는 진입 장벽이 적절한가?

### 기술 적합성

- [ ] 기존 데이터 모델과 호환되는가?
- [ ] API 하위 호환성이 유지되는가?
- [ ] 대량 데이터 (10K+) 상황에서도 동작하는가?
- [ ] 보안 원칙 (RBAC, 데이터 격리)을 준수하는가?

### 확장성

- [ ] 플러그인으로 커스터마이징할 수 있는 확장 포인트가 있는가?
- [ ] 새 어노테이션 타입/데이터 유형 추가에 유연한가?
- [ ] 국제화 (i18n) 고려가 되어 있는가?

## 제품 우선순위 프레임워크

### 기능 우선순위 결정 기준

```
1순위: 어노테이션 품질/효율성 향상
  └─ 라벨링 속도 개선, AI 보조 강화, 검수 자동화

2순위: 데이터 파이프라인 완결성
  └─ 업로드/익스포트 편의, 포맷 지원 확대, 스토리지 연동

3순위: 프로젝트 운영 효율
  └─ 대시보드, 작업자 관리, 통계, 튜토리얼

4순위: ML 파이프라인
  └─ 학습 편의, 모델 관리, 배포 자동화

5순위: 플러그인/SDK 생태계
  └─ 개발자 도구, API, 문서화
```

### 기능 타입별 접근

| 타입 | 접근 방식 |
|------|----------|
| 핵심 기능 추가 | 제품팀 주도, 플랫폼 내장, 모든 사용자 대상 |
| 도메인 특화 기능 | 플러그인으로 분리, 필요한 사용자만 설치 |
| 실험적 기능 | 플러그인 또는 베타로 먼저 제공, 검증 후 플랫폼 통합 |
| 인프라 개선 | 사용자 비가시적, 성능/안정성/보안 중심 |
